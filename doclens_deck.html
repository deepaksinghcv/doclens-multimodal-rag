01 — Motivation

Why
RAG?

An LLM only knows its training data — frozen, unsourced, and prone to confident fabrication on domain-specific questions.

The failure mode

Ask a vanilla chatbot “how do I clean this filter?” and it invents plausible steps — no source, no way to verify, wrong often enough to be dangerous in a manual.

Technical documents make it harder
  • Long context — hundreds of pages per manual
  • Procedural steps that must stay in order
  • Tables, figures & diagrams carry real meaning
  • Cross-page references and dense layout

RAG’s answer: inject fresh, private, citable evidence at inference time. The model shifts from recalling facts (hallucinates) to reading provided evidence (reliable, auditable).

Key insight: after ingestion the PDF is never read again — knowledge lives as (vector, text, page) rows.

Axis 2 · Query — online, per question

Invariant: query and chunks must share the same embedding model — else different space, garbage similarity.

Retrieval quality is decided before the LLM ever runs. Every stage is single-purpose and testable.

Chunking — the highest-leverage choice

Bounded above by the embedder’s 512-token limit (longer = silently truncated) and below by needing enough context to be self-contained. ~250-word windows with ~15% overlap keep procedures intact across boundaries.

Boilerplate removal

Running headers repeat across pages; content doesn’t. Normalize lines (drop digits), count page frequency, strip the repeaters — model-free & per-document.

Idempotent index

Deterministic content-hash IDs + delete-then-upsert per source → re-ingesting a PDF updates rows instead of duplicating them.

04 — Retrieval Two-stage: recall, then precision

Bi-encoder

Encodes query & chunk separately into vectors, compares by cosine. Precompute all chunks once → fast & indexable. Cost: less accurate — the two never “meet”.

Cross-encoder

Reads (query, chunk) together → one relevance score. Far more accurate, but nothing to precompute — must run per pair. Unusable over the whole corpus.

The resolution — chain them

Cheap bi-encoder casts a wide net (top-20). The expensive cross-encoder re-scores just those 20 and pulls the true answer to the top.

05 — Vector search

Approx.
≠
random

HNSW is an Approximate Nearest-Neighbor index: it walks a navigable graph instead of scanning all vectors — trading a little recall for big speed.

A variance mystery, debugged

Re-running the eval gave Recall 0.89 then 0.83. Not float noise, not the embeddings — those were bit-identical. Within one process: perfectly stable.

The cause — across processes

HNSW’s multi-threaded graph traversal explores candidates in an order that depends on thread scheduling, which differs between process launches. For chunks sitting right at the rank-5 boundary, that flips them in or out.

Fix: widen search (ef) → closer to exact, more stable and higher recall; or brute-force exact search for the eval.

Why caption-and-embed?

One unified index → one retrieval path, one eval harness. A VLM’s caption of a diagram (“SD-card slot labelled 1, rectangular with a triangle”) retrieves far better than a raw CLIP image vector for technical content.

Verified live

“Where is the SD-card slot?” → retrieves the figure chunk, routes to the vision model with the actual image attached, and answers by reading the diagram. Text queries stay on the text model.

07 — Generation

Read,
don’t
recall

The prompt is where reliability lives — narrow the model’s job to “synthesize this evidence”.

The grounding contract

System prompt: answer using ONLY the provided context; if it isn’t there, say you couldn’t find it. Plus temperature 0 — fidelity to evidence, not creativity. That single instruction is the main defense against hallucination.

Citations are programmatic

Page numbers come from the retrieved chunk’s metadata, in code — never scraped from the model’s prose. Trusting the LLM to report its source is itself a hallucination vector.

Refuses on cue

“Capital of India?” against 1,050 chunks → “couldn’t find it in the manual”. Grounding holds even under a mountain of tempting context.

09 — The decision that matters Measure → decline → re-measure → ship

Should we ship the reranker?

At n=21 (hand-authored): the reranker was neutral — even hurt a case. Conclusion: don’t ship complexity the data doesn’t justify.

At n=100 (broader, harder): the reranker gives a clear recall lift — the opposite result. So we shipped it, and wired it into the live pipeline.

Lesson: the size & breadth of your eval set can flip your conclusion. That’s exactly why the harness exists.

The habit worth stealing

Build the eval first, then let it make the call. The reranker shipped only after data justified it — visible in the git history as measure → decline → re-measure → ship.

Diagnose before you fix
  • High recall + low MRR ⇒ a ranking problem, not recall
  • Reranker “regressed” ⇒ suspect the metric, not the model
  • Run-to-run variance ⇒ traced to HNSW, not assumed noise
